              (query=goal, k=3, คืนมา 3 chunks ทุกครั้งจาก manual_test.txt ที่
              ingest ไว้ตั้งแต่ W3) ไม่ crash ทั้ง flow — *** แต่ manual_test.txt
              ที่ ingest ไว้ตอนนี้เป็นเนื้อหาทั่วไป (วิธี login GitHub/ค้นหา Google/
              เปิด Facebook) ไม่เกี่ยวกับ saucedemo เลย — รันจริงพิสูจน์แค่ว่า
              "ระบบเชื่อมต่อกันได้ไม่พัง" ไม่ได้พิสูจน์ว่า "RAG context ช่วยให้ agent
              ทำงานแม่นขึ้นจริง" อันนั้นต้องรอ manual ที่เนื้อหาตรงกับเว็บทดสอบจริงๆ
              ก่อนถึงจะวัดผลได้ ***

              *** W6[A] (Planner prompt: รวม page state + คู่มือ + ความจำ) ยังไม่
              ปิดจ็อบทั้งหมด — ทำแค่ส่วนคู่มือ (นี่คือ W6[B]) ส่วน "ความจำ" ยังไม่ได้
              ทำเพิ่มอะไรใหม่: ShortTermMemory (memory.py) มีอยู่แล้วแต่แค่บันทึก
              history ไว้โชว์ตอนจบ task เฉยๆ ไม่เคยถูกป้อนกลับเข้า prompt เลย —
              สิ่งที่ทำหน้าที่ "ความจำ" จริงๆ ตอนนี้คือ messages ที่ยาวขึ้นเรื่อยๆ ใน
              บทสนทนาเดียวกันของ 1 task run (LLM เห็น action/ผลลัพธ์ก่อนหน้าทั้งหมด
              ผ่าน conversation history อยู่แล้วโดยธรรมชาติ) — short/long-term
              memory แบบข้าม task จริงๆ (กันทำซ้ำ, จำ pattern) ยังเป็นของ W7 ตามแผน
              เดิม ยังไม่ได้แตะ ***

              เพิ่ม max_steps default + robustness แก้ 3 จุด (2026-07-13):
              user ขอเพิ่ม max_steps default จาก 15 เป็น 20
              (Orchestrator.run_task()) ให้ agent มี step ให้ลองมากขึ้นก่อนหมด
              โควต้า — test เดิมทุกเคสส่ง max_steps ตรงๆ อยู่แล้ว ไม่มีเคสไหนพึ่ง
              default เดิม เลยไม่ต้องแก้ test ไหนเลย

              step เยอะขึ้น = เรียก LLM API ถี่ขึ้น เลยแก้ 3 จุดกันปัญหาที่มากับ
              โควต้า/rate limit ไปด้วย:

              1) llm.py::next_action_gemini() ดัก google.api_core.exceptions.
                 ResourceExhausted (Gemini 429 quota เต็ม) ด้วย retry loop
                 (_GEMINI_RATE_LIMIT_RETRIES=3) + exponential backoff
                 (_GEMINI_RATE_LIMIT_BACKOFF_SECONDS=20 คูณ attempt คือ 20s/40s/
                 80s) แทนที่จะปล่อยให้ exception หลุดออกไป crash ทั้ง process
                 กลางลูป — ถ้า retry ครบแล้วยังโดนอยู่ ค่อย raise จริงออกไป (ไม่
                 กลืน error เงียบๆ จนวนไม่รู้จบ) unit test ใหม่ 2 เคสใน
                 test_llm.py (retry แล้วสำเร็จ, retry ครบแล้วยัง raise)

              2) orchestrator.py per-step loop เพิ่ม await asyncio.sleep(
                 _STEP_PACING_DELAY_SECONDS=3) ท้ายทุก step ที่ยังวนต่อ กันยิง
                 LLM API ถี่เกิน free-tier quota ต่อนาที (RPM) ตั้งแต่ต้น ไม่ใช่
                 แค่ดักตอน error เกิดแล้ว — ค่า 3 วิเป็น heuristic คร่าวๆ ไม่ได้
                 ผูกกับ quota จริงเป๊ะๆ ของ key ไหน (แต่ละ key/โมเดลจำกัดไม่
                 เท่ากัน) test_orchestrator.py เพิ่ม autouse fixture mock
                 asyncio.sleep (แบบเดียวกับที่ test_actions.py ทำกับ
                 _ACTION_RETRY_DELAY_SEC) กันไม่ให้ทุก test ที่วนหลาย step ต้องรอ
                 จริงเป็นวินาทีๆ

              3) เพิ่มกติกาใหม่ใน SYSTEM_PROMPT: ห้ามใช้ go_back ย้อนกลับไปหน้า
                 login หลังจาก login + เพิ่มสินค้าลงตะกร้าสำเร็จแล้ว ให้โฟกัสเดิน
                 หน้าไปหน้าตะกร้า/checkout แทน — กัน agent วน go_back กลับไปหน้า
                 login ซ้ำๆ จนติด infinite loop (เคสที่ user สังเกตเจอ) *** ยังไม่
                 เคยรันจริงผ่าน LLM ยืนยันว่ากติกาใหม่ช่วยจริง มีแค่ unit test ที่
                 ยืนยันว่า SYSTEM_PROMPT ไม่พังจากการแก้ข้อความ (ตาม pattern เดิม
                 ของกติกาอื่นๆ ที่เพิ่มไปก่อนหน้า) ***

              รวม test suite 42 เคสใน test_llm.py + test_orchestrator.py ผ่านหมด
              (แยกรันเฉพาะ 2 ไฟล์นี้เร็ว ~14 วิ พิสูจน์ว่า mock asyncio.sleep ได้
